## 20. Cloudflare Tunnel 공개 방식

현재 개발/테스트 공개 구조는 다음과 같다.

```text
Mac mini
→ Flask 서버 실행
→ Cloudflare Tunnel 연결
→ 외부 HTTPS 주소 생성
→ 다른 PC/사용자 접근 가능
```

현재 프로젝트 기준 서버 실행 위치:

```bash
cd ~/git/dev/projects/Tilon/notion_pdf
source .venv/bin/activate
python app.py
```

기본 로컬 접속 주소:

```text
http://127.0.0.1:5000
```

Cloudflare Tunnel 실행:

```bash
cloudflared tunnel --url http://127.0.0.1:5000
```

성공하면 다음 형식의 임시 외부 HTTPS 주소가 표시된다.

```text
https://xxxxx.trycloudflare.com
```

이 주소를 다른 PC, Windows 노트북, 외부 사용자에게 전달하면 현재 Mac 개발 서버에 접근할 수 있다.

주의사항:

- Flask 서버 터미널이 종료되면 외부 접속이 불가능하다.
- `cloudflared` 터미널이 종료되면 외부 접속이 불가능하다.
- `trycloudflare.com` 주소는 임시 주소이며 재실행 시 바뀔 수 있다.
- Mac 절전 모드 진입 시 연결이 끊길 수 있다.
- 현재 구조는 개발/테스트 목적이다.
- OCR 및 고화질 PDF 생성은 CPU/RAM 사용량이 크다.

상세 문서는 `docs/배포가이드.md`에 기록했다.

## 21. OCR 구조 최신 상태

현재 OCR은 속도보다 정확도와 안정성을 우선한다.

기존 방식:

```text
최종 1페이지 PDF
-> ocrmypdf 단일 후처리
```

최신 방식:

```text
Playwright screenshot clip
-> 최종 고해상도 PNG 생성
-> 하단 crop
-> 좌우 여백 보정
-> img2pdf로 1페이지 PDF 생성
-> 최종 PNG를 높이 기준 chunk로 분할
-> chunk별 OCR 전처리
-> Tesseract TSV OCR
-> OCR 좌표를 1페이지 PDF 좌표로 변환
-> invisible text layer 추가
-> pypdf로 1페이지 유지 검증
```

OCR 언어:

```text
kor+eng
```

chunk OCR 기준:

- 짧은 문서는 chunk 1개로 OCR한다.
- 긴 문서는 높이 기준 여러 chunk로 나눠 OCR한다.
- 기본 chunk 높이는 `OCR_CHUNK_HEIGHT_PX=3000`이다.
- chunk 경계 누락을 줄이기 위해 `OCR_CHUNK_OVERLAP_PX=80` overlap을 사용한다.
- OCR 실패 구간은 `ocr_failed_chunks`에 chunk 번호와 위치를 기록한다.

OCR 전처리:

- grayscale
- contrast enhancement
- sharpen
- threshold/binarize

동적 선택:

- 각 chunk에 여러 전처리 후보를 적용한다.
- 추출 텍스트 길이가 가장 좋은 전처리 결과를 선택한다.
- 코드/명령어가 많은 문서에서 과한 threshold가 항상 유리하지 않으므로 sharp/contrast 결과를 우선 비교한다.
- OCR 결과가 부족하면 다른 전처리 후보로 재시도한다.

OCR 결과 검증:

- OCR 적용 후 pypdf로 추출 가능한 텍스트 길이를 검사한다.
- 추출 텍스트가 기준보다 너무 적으면 `ocr_status=insufficient_text`로 기록한다.
- OCR 실패 또는 부족 상태여도 PDF 생성 자체는 실패하지 않는다.
- 최종 PDF는 계속 1페이지다.

현재 확인된 한계:

- OCR 텍스트 레이어는 선택/복사를 가능하게 하지만, OCR 엔진 특성상 일부 코드 문자는 오인식될 수 있다.
- 예: `-s`가 `-5`로 인식되거나, 코드 라인에 공백이 추가될 수 있다.
- 이미지처럼 렌더링된 문서 특성상 원본 HTML 텍스트를 직접 보존하는 수준의 완전한 정확도는 보장할 수 없다.
- 정확도는 문서 폰트, 글자 크기, 배경 대비, 코드블록 스타일, 이미지 압축 상태에 영향을 받는다.

유지 조건:

- `page.pdf()`는 사용하지 않는다.
- 긴 1페이지 PDF 생성 구조를 유지한다.
- 하단 crop을 유지한다.
- 좌우 여백 보정을 유지한다.
- scale 2/3 고화질 캡처를 유지한다.

## 22. 텍스트 레이어 구조 최신 상태

현재 텍스트 레이어는 OCR만 사용하지 않고, 가능한 경우 DOM 원본 텍스트를 우선 사용한다.

텍스트 레이어 모드:

```text
none   : 텍스트 레이어 없음
ocr    : OCR 기반 텍스트 레이어
dom    : DOM 원본 텍스트 기반 텍스트 레이어
hybrid : DOM 원본 텍스트 + OCR 보조
```

기본 권장 모드:

```text
hybrid
```

DOM 기반 처리:

1. Playwright로 Notion 페이지 또는 HTML 파일을 렌더링한다.
2. DOM text node를 순회한다.
3. 각 text node의 원본 `textContent`를 수집한다.
4. `Range.getClientRects()`와 `getBoundingClientRect()` 기준 위치를 수집한다.
5. computed style에서 font size, font weight, line height를 수집한다.
6. screenshot scale, crop left, paste x, rebalance shift 값을 이용해 최종 PNG 좌표로 변환한다.
7. 최종 1페이지 PDF 좌표로 다시 변환한다.
8. pypdf로 invisible text layer를 추가한다.

수집 정보:

```text
textContent
x
y
width
height
font size
font weight
line height
```

Hybrid 처리:

- DOM 텍스트는 원본 문자열이므로 영어 대문자, Windows 경로, PowerShell 명령어, 한글+영문 혼합 문장에 우선 적용한다.
- OCR은 이미지 안의 텍스트, 캡처 이미지, 표 이미지처럼 DOM text node로 잡히지 않는 영역에 보조로 적용한다.
- OCR word의 중심점이 DOM 텍스트 박스와 겹치면 중복 삽입을 피한다.

기대 효과:

- `MasterVD`, `MasterVD2`, `HOST01` 같은 대소문자 포함 문자열 복사 정확도 개선
- `C:\ClusterStorage` 같은 Windows 경로 복사 정확도 개선
- `$env:PORT=5055`, `python -m unittest discover -s tests` 같은 명령어 복사 정확도 개선
- `프로비전가능` 같은 한글 단어가 OCR처럼 이상하게 띄어쓰기되는 문제 개선

현재 한계:

- DOM 좌표 매핑은 crop/recenter를 반영하지만, 브라우저 렌더링과 PDF 텍스트 선택 엔진 차이로 선택 하이라이트 위치가 완전히 일치하지 않을 수 있다.
- DOM에 없는 이미지 내부 텍스트는 OCR에 의존한다.
- OCR 보조 영역은 여전히 OCR 엔진 특성상 일부 오인식이 있을 수 있다.
- 하이브리드 모드는 DOM 텍스트와 OCR 텍스트가 모두 들어가므로 추출 텍스트 길이가 DOM 단독보다 길어질 수 있다.

유지 조건:

- 최종 PDF는 계속 1페이지다.
- PNG 기반 PDF 생성 구조는 유지한다.
- 하단 crop을 유지한다.
- 좌우 여백 보정을 유지한다.
- 고화질 scale 2/3 캡처를 유지한다.
- `page.pdf()`는 사용하지 않는다.
