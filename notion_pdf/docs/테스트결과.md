## 2026-05-16 - DOM text layer comparison test

### 구현 확인

- PNG 캡처 기반 1페이지 PDF 생성 구조는 유지했다.
- `page.pdf()`는 사용하지 않는다.
- 텍스트 레이어 모드를 추가했다.
  - `none`: 텍스트 레이어 없음
  - `ocr`: OCR 기반 텍스트 레이어
  - `dom`: DOM 원본 텍스트 기반 텍스트 레이어

- Playwright 렌더링 후 DOM text node에서 다음 정보를 수집한다.
  - `textContent`
  - `getBoundingClientRect()`
  - font size
  - font weight
  - line height
  - x, y, width, height
- crop/recenter 이후의 이미지 좌표 변환값을 이용해 DOM 좌표를 최종 1페이지 PDF 좌표로 매핑한다.

## 2026-05-16 - v3.1 품질 저하 원인 분석 및 복구

### v3.0 대비 변경사항 분석

`git diff v3.0 -- app.py` 기준 주요 차이:

- DOM 텍스트 수집 방식이 v3.0의 `TreeWalker` text node 기반에서 block/line 기반으로 변경됐었다.
- OCR 단독 모드까지 텍스트 레이어 삽입 전에 좌표 정렬을 강제하면서 v3.0의 Tesseract TSV 출력 순서가 바뀌었다.
- 기본 텍스트 레이어 노출이 `DOM + OCR` 중심으로 유지돼 실제 PDF 뷰어에서 불안정한 hybrid 복사 결과가 먼저 노출됐다.
- Hybrid 모드에서 DOM과 OCR이 같은 영역에 중복 삽입될 가능성을 줄이기 위한 마스킹은 있었지만, 실제 뷰어 드래그 선택에서는 DOM/OCR 레이어가 섞일 수 있었다.

### 현재 처리 흐름

1. Playwright로 HTML/URL을 렌더링한다.
2. DOM 높이, content box, DOM text node 좌표를 수집한다.
3. 긴 전체 화면 PNG를 고해상도로 캡처한다.
4. 하단 crop, 좌우 여백 보정, 최종 1페이지 PDF 이미지를 생성한다.
5. 텍스트 레이어 옵션에 따라 다음 중 하나를 적용한다.
   - `none`: 텍스트 레이어 없음
   - `ocr`: OCR TSV 결과만 invisible text layer로 삽입
   - `dom`: DOM 원문 텍스트만 invisible text layer로 삽입
   - `hybrid`: DOM 원문을 먼저 삽입하고, DOM 영역과 겹치지 않는 OCR만 보조 삽입

### 원인

- block/line 기반 DOM 수집은 일부 Notion 구조에서 span/div text node를 놓칠 수 있어 선택/복사 가능한 영역이 줄어들 수 있다.
- OCR-only 모드에서 좌표 정렬을 강제하면 Tesseract가 출력한 읽기 순서를 바꾸므로 v3.0보다 복사 순서가 나빠질 수 있다.
- DOM + OCR은 pypdf 추출에서는 좋아 보여도 실제 PDF 뷰어 드래그 선택에서 두 invisible text layer가 같이 잡히면 문장이 섞일 수 있다.

### 복구 내용

- DOM 수집은 v3.0의 `TreeWalker` text node 기반으로 되돌렸다.
- OCR-only 모드는 v3.0처럼 Tesseract TSV 입력 순서를 보존하도록 되돌렸다.
- DOM-only는 DOM 항목만 정렬하고, Hybrid는 DOM을 먼저 넣은 뒤 DOM 영역과 겹치지 않는 OCR만 뒤에 넣도록 조정했다.
- 이후 안정화 정책에 따라 기본 텍스트 레이어 옵션은 `none`으로 변경했다.
- DOM과 DOM + OCR은 옵션으로 유지하되 기본값에서는 제외했다.

- v3.0 안정 PDF 생성 기능은 다시 정상 동작한다.
